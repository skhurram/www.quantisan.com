Abstracting away broker trading APIs

Retail trading brokerage that offers a programming interface typically allow connections through their proprietary API. The advantage is that basic functionality such as price data stream handling and accounting are all built into the API client. The disadvantage is that using a framework is another layer of implementation complexity. This is manageable for a single-broker trading system because you're always using with the same framework. However, I was building a trading system that needs to interact with multiple brokers at runtime. I have no intention of juggling multiple frameworks at a time. A solution that I employ is to only use each API for basic functions and abstract away the implementations with a message-oriented interface.

Fundamentally, there are three tasks that you interact with a broker for.

  1. Get price data
  2. Execute order
  3. Get account information
