Managing data workflow development with continuous integration

Our company data workflow consists of a development data pipeline sinking to a development database and a production pipeline sinking to a production database. Immutable source data are stored on Amazon S3. We periodically run a stable release and a bleeding edge development version of our Cascalog data processing jar. Each is sinking to a corresponding production and development databases on Amazon Redshift. We break things on the development side and leave the production side intact for our analysts to do their magic. This might sound like a job description for a deployment manager but both branches of our build, test, deploy, and execute process happens automatically using continuous integration right from the moment of a git push. We don't even need to specify which deploy branch to use. This post will show how that happens.

A trivial solution is to keep two databases. One for production, one for development. 

[Continuous Integration](http://martinfowler.com/articles/continuousIntegration.html) is a software development practice where members of a team integrate their work frequently, usually each person integrates at least daily - leading to multiple integrations per day. Each integration is verified by an automated build (including test) to detect integration errors as quickly as possible.
