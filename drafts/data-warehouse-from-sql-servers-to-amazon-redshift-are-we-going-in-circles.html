Data warehouse from SQL Servers to Amazon Redshift, are we going in circles?

After ditching our SQL servers two years ago, we are coming back in circle nowadays sinking processed data on Hadoop into Amazon Redshift with dimensional modelling techniques. From the analyst's perspective, they're able to use their ODBC/JDBC tools again and use SQL to query data. In addition to the whole data process being simpler and cheaper, what has changed from a purely business intelligence standpoint is the increase in degree-of-freedom of the data that people have access to.

A couple years ago, the accessible data at uSwitch were all aggregates of some sort, visitor counts, transaction numbers, etc. We could drill down the data to different levels by breaking these metrics into sources of traffic or demographic information. Still, there is only so much you can slice and dice our datasets because we were working on curated material. So when a product manager wants to ask novel questions beyond the scope of the data dimensional model, we needed to hack into our data warehouse on SQL servers through this ETL process:

When doing anything new with the data required going through that, you simply cannot do serendipitous exploration. Rebuilding our data workflow and changing our notion of data was one of the first problems we tackled back in early 2012.

For a while we considered that problem solved with Kafka for collection, Hadoop for storage, Cascalog for ETL, and Hive. Thus enabling us to decouple data storage, data processing, and data analytics. We were able to freely and quickly interrogate our down to the individual person and event.

In summary, what we have gained are:

  1. Apply software engineering techniques to processing data
    • separation of concerns in data processing
    • source code versioning
    • modules are in continuous integration workflow to ensure catching breaking changes early
  2. Separation of data infrastructure
    • collection
    • processing
    • querying

Processing and manipulating data isn't new. Some people call it database administration. I call it computer science.
